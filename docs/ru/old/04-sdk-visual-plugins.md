# Разработка визуальных плагинов (C++ SDK)

Визуальный плагин Shader Desk — это динамически загружаемая библиотека (`.so`), работающая в адресном пространстве графического ядра. Плагин получает готовый EGL-контекст (OpenGL ES 3.0), точное значение `delta_time` и прямой доступ к разделяемой памяти (BlackBoard) для чтения системных метрик с нулевой задержкой.

В этом разделе описан процесс создания нативных визуальных эффектов с использованием официального C++ SDK, иерархия классов, паттерны проектирования и строгие правила, которые необходимо соблюдать для стабильной работы в составе микроядра.

---

## 1. Архитектура плагина и языковая независимость

Интерфейс плагинов (`IWallpaperEffectABI`) построен на **строгом C-ABI** без использования C++ STL в публичных методах. Подробное описание причин и механизмов приведено в разделе [Архитектура: Плагины и C-ABI](07-architecture-overview.md#5-плагины-и-интерфейс-песочных-часов-c-abi).

Ключевые следствия для разработчика:

- Все публичные методы интерфейса принимают и возвращают только POD-типы и сырые указатели.
- Параметры передаются через структуру `ParamValueABI` фиксированного размера (512 байт).
- Плагин обязан экспортировать функцию `get_abi_version()`.

Благодаря этой архитектуре вы можете писать плагины не только на C++, но и на Rust, Zig или чистом C — при условии корректного экспорта C-совместимых функций. Однако официальный SDK и инструмент генерации кода ориентированы на максимальное удобство именно в C++17/20.

## 2. Иерархия классов SDK

Для разработки визуальных плагинов SDK предоставляет два основных базовых класса, реализующих разные подходы к рендерингу.

### 2.1. WallpaperEffect (Стандартный конвейер)

Базовый класс для большинства 2D-эффектов и пост-обработки. Предоставляет:

- **Zero-VBO Fullscreen Triangle** — полноэкранный треугольник, генерируемый на GPU через `gl_VertexID`. Не требует создания VBO на CPU.
- **Автоматическое управление временем** — переменная `time` обновляется в `render()` и передаётся в шейдер как униформа.
- **Масштабирование FBO** — встроенная поддержка параметра `layer_fbo_scale` для рендеринга в пониженном разрешении.
- **Вложенные фильтры** — автоматическое применение цепочки пост-эффектов через `set_filters_abi()`.
- **Shadow-Commit** — встроенный механизм безопасной перезагрузки шейдеров.

Плагин, наследующий `WallpaperEffect`, переопределяет только методы `get_parameters()`, `set_parameter()`, `initialize()` и хук `bind_custom_uniforms()`. Метод `render()` переопределять не требуется, если используется стандартный конвейер.

### 2.2. KinematicEffect (Физика и глобальная камера)

Наследник `WallpaperEffect`, добавляющий поддержку 3D-физики и интеграцию с глобальной камерой сцены:

- **Инерция и угловая скорость** — автоматическое применение трения (`rotation_decay`) к вращению.
- **Центр масс (Pivot)** — возможность задавать точку вращения объекта через `pivot_offset`.
- **Глобальная камера** — чтение позиции, цели и вектора вверх из BlackBoard (управляется через `core.set_camera()` в Lua).
- **Матрицы MVP** — вычисление `model_matrix`, `view_matrix`, `proj_matrix` в методе `update_kinematics()`.

Плагин, наследующий `KinematicEffect`, вызывает `init_kinematics()` в `initialize()` и `update_kinematics()` в начале своего `render()`.

## 3. Быстрый старт: генерация кода из шейдера

Для создания нового плагина не требуется писать шаблонный C++ код вручную. Утилита `shader-desk-generate` парсит GLSL-файл и генерирует готовый класс.

**Шаг 1: Подготовка шейдера**

Создайте директорию плагина и папку `shaders/`. Внутри разместите фрагментный шейдер с аннотациями:

```glsl
// shaders/my_effect_frag.glsl
#version 300 es
precision highp float;

// @param speed | float | 1.5 | Скорость анимации
// @param base_color | vec3 | 0.8, 0.2, 0.5 | Основной цвет
// @texture u_texture | default.png | Текстурная карта

uniform float speed;
uniform vec3 base_color;
uniform sampler2D u_texture;

out vec4 FragColor;
void main() {
    // Ваш код...
    FragColor = vec4(base_color, 1.0);
}
```

Поддерживаемые типы: `float`, `int`, `bool`, `vec2`, `vec3`, `vec4`, `string`. Аннотация `@texture` автоматически генерирует код для загрузки изображения через `stb_image`.

**Шаг 2: Генерация**

Находясь в корневой директории плагинов, выполните:

```bash
shader-desk-generate my-custom-effect
```

Утилита создаст:
- `CMakeLists.txt` — готовый к сборке.
- `my-custom-effect.hpp` — заголовочный файл класса.
- `my-custom-effect.cpp` — реализация с хуками для Uniform-ов.
- `presets/default.lua` — файл конфигурации по умолчанию.

Генератор автоматически связывает GLSL-параметры с C++-переменными и вызовами `glUniform*`.

## 4. Стандартный конвейер (WallpaperEffect)

### 4.1. Инициализация

Метод `initialize()` должен вызвать `init_standard_pipeline()`, которая загружает и компилирует шейдеры:

```cpp
bool MyEffect::initialize(ICoreContext* core, uint32_t width, uint32_t height) {
    bool success = init_standard_pipeline(core, "my_effect_vert.glsl", "my_effect_frag.glsl");
    if (success) {
        // Получение локаций Uniform-ов
        u_custom = glGetUniformLocation(m_std_program, "custom_param");
    }
    return success;
}
```

Если имя вершинного шейдера передано пустой строкой, используется встроенный Zero-VBO шейдер.

### 4.2. Передача Uniform-ов

Вместо переопределения `render()` используется хук `bind_custom_uniforms()`:

```cpp
void MyEffect::bind_custom_uniforms() {
    GLuint program = m_std_program;
    glUniform1f(u_custom, custom_value);
}
```

Этот метод вызывается автоматически внутри `render()` после `glUseProgram()` и перед `glDrawArrays()`.

### 4.3. Безопасная перезагрузка шейдеров

Для горячей замены шейдера (например, при изменении `shader_theme` из Lua) используется метод `request_shader_reload()`:

```cpp
if (name == "shader_theme") {
    std::string new_theme = std::get<std::string>(value);
    if (new_theme != active_shader) {
        active_shader = new_theme;
        request_shader_reload(active_shader + "_frag.glsl");
    }
}
```

Базовый класс гарантирует безопасность: при ошибке компиляции старый шейдер продолжает работать, а в терминал выводится сообщение.

### 4.4. Работа с текстурами

Аннотация `@texture` генерирует методы загрузки изображений. В C++-коде доступны:

- Переменные: `texture_id`, `texture_path`, `texture_w`, `texture_h`.
- Флаг `texture_pending = true` для отложенной загрузки.
- Метод `process_textures()`, который автоматически вызывается в `bind_custom_uniforms()`.

Если требуется ручная загрузка, используйте `stb_image.h` (включается автоматически при наличии `@texture`).

## 5. Физика и кинематика (KinematicEffect)

### 5.1. Инициализация физики

В `initialize()` вызовите `init_kinematics(core)`:

```cpp
bool MyEffect::initialize(ICoreContext* core, uint32_t width, uint32_t height) {
    init_kinematics(core);
    // Остальная инициализация...
}
```

Это свяжет плагин с глобальной камерой и BlackBoard.

### 5.2. Обновление состояния

В начале `render()` вызовите `update_kinematics(dt, aspect_ratio)`:

```cpp
void MyEffect::render(uint32_t width, uint32_t height, float dt) {
    float aspect = (height > 0) ? (float)width / height : 1.0f;
    update_kinematics(dt, aspect);
    // После этого доступны model_matrix, view_matrix, proj_matrix
}
```

### 5.3. Управление физикой через Lua

Базовый класс автоматически регистрирует параметры: `offset`, `rotation_axis`, `rotation_speed`, `rotation_decay`, `pivot_offset`. В `set_parameter()` используйте `set_kinematic_param()` для их обработки:

```cpp
void MyEffect::set_parameter(const std::string& name, const EffectParameterValue& value) {
    if (set_kinematic_param(name, value)) return;
    // Обработка локальных параметров...
}
```

### 5.4. Глобальная камера

Параметры камеры управляются из Lua через `core.set_camera()` и автоматически применяются ко всем наследникам `KinematicEffect`. Это позволяет синхронизировать движение камеры между разными 3D-объектами на сцене.

## 6. Вложенные фильтры (Nested Filters)

Визуальный плагин может выступать как хост для цепочки пост-эффектов. Ядро передаёт массив фильтров через ABI-метод `set_filters_abi()`.

SDK автоматически управляет этой цепочкой внутри метода `render()`:

1. Основной объект рендерится во внутренний FBO.
2. В `end_scaled_pass()` фильтры применяются последовательно с использованием Ping-Pong-буферов.
3. Финальный результат композитится в основной FBO ядра.

Это позволяет создавать сложные конвейеры, где один эффект может модифицировать результат другого (например, глитч-фильтр поверх 3D-сферы). Разработчику не требуется вручную управлять FBO или текстурами — всё обрабатывается базовым классом.

---

## 7. Масштабирование и изоляция FBO

Ключевой механизм оптимизации и изоляции — это пара методов `begin_scaled_pass()` и `end_scaled_pass()`.

### 7.1. begin_scaled_pass

Вызывается в начале `render()`. Проверяет:

- Если `layer_fbo_scale == 1.0` и нет вложенных фильтров, возвращает текущие размеры FBO без изменений (pass-through).
- В противном случае создаёт или изменяет размер внутреннего FBO, копирует Z-буфер из основного FBO и перенаправляет все последующие Draw Calls во внутренний буфер.

### 7.2. end_scaled_pass

Выполняет обратное преобразование:

- Блиттинг Z-буфера из внутреннего FBO обратно в основной (сохранение 3D-корректности).
- Последовательное применение цепочки фильтров (если они есть) через Ping-Pong.
- Финальный блиттинг цветового буфера с применением альфа-композитинга.

Этот подход обеспечивает:

- Экономию производительности (рендеринг сложных шейдеров в половинном разрешении).
- Полную изоляцию стейта плагина от других слоёв.
- Корректное наложение 3D-объектов с учётом глубины (Z-Buffer).

### 7.3. Схема работы FBO-конвейера

```mermaid
flowchart TD
    A[Основной FBO ядра] -->|begin_scaled_pass| B{scale != 1.0<br>или есть фильтры?}
    B -->|Нет| C[Рендеринг напрямую в основной FBO]
    B -->|Да| D[Создание/изменение размера<br>внутреннего FBO[0] и FBO[1]]
    D --> E[Копирование Z-буфера<br>из основного FBO во внутренний]
    E --> F[Рендеринг объекта<br>во внутренний FBO[0]]
    F --> G[end_scaled_pass]
    G --> H{Есть фильтры?}
    H -->|Нет| I[Блиттинг цвета из<br>внутреннего FBO в основной]
    H -->|Да| J[Цикл по фильтрам<br>с Ping-Pong переключением]
    J --> K[Финальный фильтр<br>пишет в основной FBO]
    I --> L[Восстановление стейта]
    K --> L
    C --> L
```

## 8. Контракт с ядром (Строгие правила)

Ваш плагин встраивается в сложный конвейер, который может включать многослойный рендеринг и пост-обработку. Для корректной работы соблюдайте следующие правила.

### 8.1. Zero-Allocation Loop (запрет выделения памяти в render())

Метод `render()` является критической секцией Zero-Latency-цикла Wayland и профилируется макросами Tracy Profiler.

Внутри `render()` **категорически запрещено**:

- Вызывать `new`, `malloc`, `calloc`, `realloc`.
- Создавать объекты STL-контейнеров, требующие динамической памяти (`std::vector`, `std::string`, `std::map`).
- Вызывать `std::shared_ptr` или `std::unique_ptr` с выделением.
- Неявно создавать строки через конкатенацию (`std::string s = "a" + "b";`).

Вся память, векторы и матрицы должны быть преаллоцированы один раз на этапе `initialize()`. Любая аллокация в `render()` будет зафиксирована телеметрией как архитектурное нарушение.

### 8.2. Изоляция состояния OpenGL

Плагин не имеет права «мусорить» в глобальном стейте контекста:

- Если вы используете VAO, в конце `render()` вызовите `glBindVertexArray(0)`.
- Если вы отключаете тест глубины (`glDisable(GL_DEPTH_TEST)`), вы обязаны вернуть его в исходное состояние (`glEnable(GL_DEPTH_TEST)`) до выхода.
- То же самое касается режимов смешивания (`glBlendFunc`), вьюпорта и активных текстурных юнитов.

Нарушение этого правила приведёт к артефактам в соседних слоях и сложно отлаживаемым багам.

### 8.3. Запрет на очистку кадра

Не вызывайте `glClear()` и `glViewport()` внутри `render()`. Ядро самостоятельно:

- Биндит нужный Ping-Pong FBO.
- Устанавливает Viewport под физическое разрешение дисплея.
- Очищает буферы перед вызовом первого слоя.

Если вашему плагину требуется очистить только Z-буфер (для корректной работы 3D-сцены), используйте флаг `clear_depth = true` в конфигурации слоя.

### 8.4. Экспорт ABI версии (Обязательно)

Ваша динамическая библиотека обязана экспортировать функцию `get_abi_version()`:

```cpp
extern "C" {
    uint32_t get_abi_version() {
        return SHADER_DESK_ABI_VERSION;
    }
    IWallpaperEffectABI* create_effect() {
        return new MyEffect();
    }
    void destroy_effect(IWallpaperEffectABI* effect) {
        delete static_cast<MyEffect*>(effect);
    }
}
```

Плагин-менеджер ядра сверяет это значение перед загрузкой. Если функция отсутствует или версии не совпадают, ядро откажется загружать `.so`.

### 8.5. Корректная обработка on_cleanup()

Метод `on_cleanup()` вызывается при уничтожении плагина. В нём необходимо освободить все собственные ресурсы OpenGL (текстуры, VBO, VAO). Базовый класс автоматически удаляет `m_std_program` и `m_std_vao`, поэтому удалять их вручную не нужно.

Важно: перед вызовом `glDelete*` проверяется наличие активного EGL-контекста. Если контекст утерян (например, после выхода из спящего режима), вызовы `glDelete*` пропускаются во избежание краша драйвера.
