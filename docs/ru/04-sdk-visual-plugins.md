# Разработка визуальных плагинов (C++ SDK)

Визуальный плагин Shader Desk — это динамически загружаемая библиотека (`.so`), работающая в адресном пространстве графического ядра. Плагин получает готовый EGL-контекст (OpenGL ES 3.0), точное значение `delta_time` и прямой доступ к разделяемой памяти (BlackBoard) для чтения системных метрик с нулевой задержкой.

## 1. Архитектура и языковая независимость (C-ABI)

Для гарантии бинарной совместимости между ядром и плагинами, скомпилированными разными версиями компиляторов, используется паттерн **Hourglass Interface**.

1. **Нижний уровень (C-ABI):** Взаимодействие происходит исключительно через интерфейсы из `plugin-abi.hpp` (`IWallpaperEffectABI`). Они содержат только чисто виртуальные методы и POD-структуры фиксированного размера. Передача STL-контейнеров (`std::string`, `std::vector`) через границу библиотеки строго запрещена.
2. **Верхний уровень (C++ SDK):** Для удобной разработки предоставляется базовый класс `WallpaperEffect` (`wallpaper-effect.hpp`). Он реализует все `_abi` методы под капотом, скрывает работу с сырыми указателями и предоставляет идиоматичный C++ API с использованием `std::variant`.

**Разработка на других языках:**
Поскольку интерфейс плагинов основан на строгом C-ABI, использование C++ **не является обязательным**. Вы можете написать плагин на любом языке, способном экспортировать C-совместимые функции в разделяемую библиотеку (`.so`). 
Архитектура позволяет создавать плагины на **Rust** (через `#[no_mangle] extern "C"`), **Zig** или чистом **C99**. Однако официальный SDK, генераторы кода и встроенная математика (`GLM`) в данный момент ориентированы на предоставление максимального комфорта именно для современного C++ (17/20).

### Иерархия загрузки (Fallback-архитектура)

Графическое ядро использует строгий порядок поиска скомпилированных `.so` библиотек. Если плагин с одинаковым именем найден на нескольких уровнях, загружается вариант с наивысшим приоритетом:

1. **User Workspace:** `~/.config/interactive-wallpaper/effects/` (Наивысший приоритет. Позволяет модифицировать системные плагины без root-прав).
2. **Local (In-tree):** Директория `plugins/` рядом с исполняемым файлом `interactive-wallpaper` (Используется при портативной сборке).
3. **System FHS:** Системные директории. В зависимости от дистрибутива (из-за стандарта `GNUInstallDirs`) это может быть `/usr/lib/shader-desk/plugins/`, `/usr/lib64/shader-desk/plugins/` или `/usr/lib/x86_64-linux-gnu/shader-desk/plugins/`.

## 2. Быстрый старт: Автогенерация кода из шейдера

Вам не нужно писать шаблонный C++ код (boilerplate) вручную. SDK включает утилиту `shader-desk-generate`, которая парсит ваш GLSL-файл и генерирует готовый C++ класс.

**Шаг 1: Подготовка шейдера**
Создайте директорию для плагина, внутри нее папку `shaders/`, и поместите туда ваш фрагментный шейдер. Опишите параметры, которые вы хотите вынести в Lua-конфиг, с помощью специальных аннотаций `// @param`:

Синтаксис: `// @param [имя] | [тип] | [значение_по_умолчанию] | [описание]`

```glsl
// shaders/my_effect_frag.glsl
#version 300 es
precision highp float;

// @param speed | float | 1.5 | Скорость анимации
// @param enable_grid | bool | false | Отрисовка фоновой сетки
// @param base_color | vec3 | 0.8, 0.2, 0.5 | Основной цвет свечения

uniform float speed;
uniform bool enable_grid;
uniform vec3 base_color;

out vec4 FragColor;
void main() {
    // Ваш код...
    FragColor = vec4(base_color, 1.0);
}
```
*Поддерживаемые типы: `float`, `int`, `bool`, `vec3`.*

**Шаг 2: Генерация**
Находясь в директории `plugins/`, выполните команду:
```bash
shader-desk-generate my-custom-effect
```

**Результат:**
Утилита создаст `CMakeLists.txt`, заголовочный файл и `.cpp` исходник. Сгенерированный C++ код уже включает реализацию методов `get_parameters()` и `set_parameter()`, а также вызовы `glUniform*` для передачи данных в видеокарту. Код готов к сборке (через `cmake`) и отрисует ваш шейдер на полноэкранном треугольнике (Fullscreen Quad).

## 3. Расширение сгенерированного C++ кода

Автогенератор создает отличную базу для 2D-эффектов (в стиле ShaderToy). Однако истинная мощь нативного плагина раскрывается, когда вы начинаете модифицировать сгенерированный `.cpp` файл. 

Что можно и нужно реализовывать на стороне C++:

### 3.1. Создание сложной 3D-геометрии
Сгенерированный код использует подход "Zero-VBO Fullscreen Triangle". В этом режиме координаты вершин вычисляются прямо во фрагментном шейдере на основе встроенной переменной `gl_VertexID`, что избавляет от необходимости передавать данные из CPU. 

**Критическое правило OpenGL ES 3.0:** Несмотря на отсутствие VBO, стандарт (Core Profile) **строго требует** наличия привязанного объекта `VAO` (Vertex Array Object) при вызове `glDrawArrays`. Сгенерированный код создает пустой `VAO` именно для удовлетворения этого требования. Если вы удалите `glGenVertexArrays`, драйвер видеокарты (например, Mesa) немедленно завершит работу композитора с ошибкой `GL_INVALID_OPERATION`.

Если ваш эффект подразумевает 3D-объекты, удалите код пустого `VAO` из `initialize()` и создайте полноценные связки буферов (`VAO`, `VBO`, `EBO`):


```cpp
bool MyEffect::initialize(ICoreContext* core, uint32_t width, uint32_t height) {
    // ... загрузка шейдеров ...

    // Генерация меша (например, Икосаэдра или загрузка из OBJ)
    std::vector<glm::vec3> vertices = generate_geometry();
    
    glGenVertexArrays(1, &vao);
    glGenBuffers(1, &vbo);
    
    glBindVertexArray(vao);
    glBindBuffer(GL_ARRAY_BUFFER, vbo);
    glBufferData(GL_ARRAY_BUFFER, vertices.size() * sizeof(glm::vec3), vertices.data(), GL_STATIC_DRAW);
    
    glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, sizeof(glm::vec3), (void*)0);
    glEnableVertexAttribArray(0);
    // ...
}
```

### 3.2. Вычисления и физика на процессоре (CPU)
Многие алгоритмы (симуляция поведения толпы Boids, сложная орбитальная механика частиц) крайне сложно или неэффективно реализовывать внутри GLSL фрагментного шейдера.
Используйте метод `render()` для обновления стейта с использованием библиотеки `GLM`, после чего загружайте обновленные данные в VBO или передавайте через Uniform Arrays.

```cpp
void MyEffect::render(uint32_t width, uint32_t height, float dt) {
    // 1. Физика на CPU
    angular_velocity *= rotation_decay; // Применение инерции
    glm::quat rotation_delta = glm::angleAxis(glm::length(angular_velocity) * dt, glm::normalize(angular_velocity));
    orientation = glm::normalize(rotation_delta * orientation);

    // 2. Формирование матриц
    glm::mat4 model = glm::translate(glm::mat4(1.0f), offset) * glm::toMat4(orientation);
    glm::mat4 projection = glm::perspective(glm::radians(45.0f), (float)width / (float)height, 0.1f, 100.0f);

    // 3. Рендеринг
    glUseProgram(program);
    glUniformMatrix4fv(u_model, 1, GL_FALSE, &model[0][0]);
    // ... glDrawElements ...
}
```

### 3.3. Доступ к шине BlackBoard (Zero-Latency IPC)
Плагины могут читать сырые данные от системных демонов (звук, положение мыши, системные метрики) абсолютно бесплатно, минуя Lua. Шина BlackBoard хранит данные непрерывно.

В методе `initialize()` запросите и сохраните указатель на нужную ячейку памяти. Этот указатель остается валидным на протяжении всей жизни плагина.

```cpp
// В файле заголовка: float* p_mouse_x = nullptr; float* p_bass = nullptr;

bool MyEffect::initialize(ICoreContext* core, uint32_t width, uint32_t height) {
    p_mouse_x = core->get_blackboard()->bind_float("mouse.accum_x");
    p_bass    = core->get_blackboard()->bind_float("audio.bass");
    // ...
}

void MyEffect::render(uint32_t width, uint32_t height, float dt) {
    // Прямое чтение из памяти демона (O(1))
    if (p_bass) {
        float current_bass = *p_bass;
        glUniform1f(u_bass_amplitude, current_bass);
    }
}
```


## 4. Контракт с ядром (Строгие правила)

Ваш плагин встраивается в сложный конвейер, который может включать многослойный рендеринг и пост-обработку. Для корректной работы соблюдайте следующие правила:

1. **Изоляция машины состояний OpenGL:**
   Плагин не имеет права "мусорить" в глобальном стейте контекста.
   * Если вы используете VAO, в конце метода `render()` обязательно вызовите `glBindVertexArray(0)`.
   * Если вы отключаете тест глубины (`glDisable(GL_DEPTH_TEST)`) или меняете режимы смешивания (`glBlendFunc`), вы обязаны вернуть их в исходное состояние (например, `glEnable(GL_DEPTH_TEST)`) до выхода из метода.

2. **Запрет на очистку кадра:**
   Не вызывайте `glClear()` и `glViewport()`. Ядро самостоятельно биндит нужный Ping-Pong FBO, устанавливает Viewport под физическое разрешение дисплея (с учетом HiDPI и пользовательского скейлинга) и очищает буфер перед вызовом слоев.

3. **Нулевые аллокации (Zero-Allocation Loop):**
   Метод `render()` является критической секцией Zero-latency цикла Wayland и профилируется макросами `Tracy Profiler` (`ZoneScoped`). 
   Внутри этого метода **категорически запрещено** выделять память (использовать операторы `new`, `malloc`, неявное создание `std::string`, умных указателей `std::shared_ptr` или вызов `std::vector::push_back` сверх зарезервированной емкости `reserve()`).
   
   *Почему это важно:* Стандартные аллокаторы ОС используют глобальные мьютексы. В многопоточной среде Linux вызов `malloc` может заблокировать поток рендеринга на неопределенное время в ожидании освобождения мьютекса другим процессом. Кроме того, при сборке ядра с `-DENABLE_PROFILING=ON` движок глобально переопределяет операторы `new/delete` для профилирования, что добавляет дополнительный оверхед телеметрии. 
   
   Вся память, векторы и матрицы должны быть преаллоцированы один раз на этапе `initialize()`. Любая аллокация в `render()` будет зафиксирована телеметрией как грубое архитектурное нарушение.

4. **Экспорт версии ABI (Обязательно для сторонних языков):**
   Помимо функций `create_effect` и `destroy_effect`, ваша динамическая библиотека обязана экспортировать функцию `get_abi_version()`. Плагин-менеджер ядра сверяет это значение перед вызовом конструкторов. Если функция отсутствует или версии не совпадают, ядро откажется загружать `.so` файл в память для предотвращения бинарных крашей (Segfaults).
   
   *Пример экспорта (C++):*
   ```cpp
   extern "C" {
       uint32_t get_abi_version() { return SHADER_DESK_ABI_VERSION; }
       IWallpaperEffectABI* create_effect() { return new MyEffect(); }
       void destroy_effect(IWallpaperEffectABI* effect) { delete static_cast<MyEffect*>(effect); }
   }
   ```
