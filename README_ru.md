# Shader Desk

Shader Desk  это модульный, высокопроизводительный движок интерактивных обоев для Wayland. 

Проект спроектирован на базе архитектуры микроядра: рендеринг, сбор данных и визуальные эффекты строго изолированы. Тяжелые операции (например, быстрое преобразование Фурье для звука или чтение `/dev/input`) вынесены в самостоятельные процессы (демоны). Они передают данные в графическое ядро через неблокирующие UNIX-сокеты (`epoll`, `MSG_DONTWAIT`), что гарантирует нулевую задержку (zero-latency) и защищает графический сеанс пользователя от зависаний.

## Совместимость с графическими окружениями

Для вывода изображения на фон рабочего стола Shader Desk использует открытый протокол **`zwlr_layer_shell_v1`**. Проект работает исключительно под управлением сеансов Wayland (X11 не поддерживается).

**Полная поддержка:**
Движок гарантированно работает на всех композиторах, реализующих `layer-shell` актуальных версий:
* Hyprland
* Sway
* KWin (KDE Plasma Wayland)
* COSMIC
* niri, river, Labwc, Wayfire, GameScope и другие композиторы на базе `wlroots`.

**Не поддерживается:**
* **GNOME (Mutter)** и **Cinnamon (Muffin)**. Данные окружения принципиально отказываются от реализации открытого стандарта `wlr-layer-shell`, из-за чего вывод обоев на уровень фона рабочего стола в них невозможен.

## Установка и сборка

Зависимости времени сборки: `cmake`, `pkg-config`.
Зависимости библиотек: `wayland-client`, `wayland-protocols`, `egl`, `glesv2`, `luajit`.
Зависимости демонов (опционально): `fftw3`, `libpulse`, `libevdev`.

### 1. Компиляция проекта
```bash
git clone https://github.com/your-username/shader-desk.git
cd shader-desk
cmake -B build-release -DCMAKE_BUILD_TYPE=Release
cmake --build build-release -j$(nproc)
```

### 2. Выбор режима использования

Архитектура Shader Desk поддерживает два независимых сценария работы:

**Вариант А: Портативный режим (Без установки)**
Вы можете запускать движок прямо из папки сборки. Ядро автоматически найдет свежие скомпилированные `.so` плагины в директории `build-release/plugins`. Это идеальный вариант для тестирования и участия в разработке (контрибьюции).
```bash
./build-release/interactive-wallpaper
```

**Вариант Б: Системная установка (FHS)**
Для повседневного использования рекомендуется установить движок в систему. Инсталляция строго следует стандарту иерархии файловой системы (FHS). Бинарные исполняемые файлы будут помещены в `/usr/bin/`, а общие ресурсы и пресеты — в `/usr/share/shader-desk/`.

**Примечание по путям библиотек:** Сборочная система использует стандартный модуль CMake `GNUInstallDirs`. В зависимости от архитектуры вашего дистрибутива (Fedora, Arch Linux, Ubuntu), скомпилированные бинарные плагины (`.so`) будут автоматически размещены в соответствующей директории системных библиотек: `/usr/lib/shader-desk/plugins/`, `/usr/lib64/shader-desk/plugins/` или `/usr/lib/x86_64-linux-gnu/shader-desk/plugins/`.

```bash
sudo cmake --install build-release
```

## Инициализация рабочего пространства и конфигурация

Shader Desk использует многоуровневую систему поиска плагинов (Fallback-архитектура). Если вы хотите изменять шейдеры или настраивать эффекты, вам необходимо создать пользовательское рабочее пространство (Sandbox).

### Шаг 1: Создание песочницы (Workspace)
Выполните следующую команду:
```bash
interactive-wallpaper --init-workspace
```
Утилита создаст директорию `~/.config/interactive-wallpaper/effects/`, скопирует туда исходные коды всех доступных плагинов (шейдеры, пресеты) и внедрит в них скомпилированные `.so` бинарники. 

Теперь вы можете безопасно редактировать любые `.glsl` файлы внутри `~/.config/`. Движок будет отслеживать изменения и мгновенно перекомпилировать шейдеры "на лету", не затрагивая системные файлы в `/usr/`.

### Шаг 2: Генерация конфигурации
Для автоматического сканирования плагинов и создания файлов настроек выполните:
```bash
interactive-wallpaper --init-config
```
Менеджер плагинов просканирует файловую систему, используя архитектуру Fallback-приоритетов. Если плагин обнаружен в изолированном пользовательском рабочем пространстве (`~/.config/interactive-wallpaper/effects/`), движок загрузит именно его, полностью проигнорировав системную версию из `/usr/lib/` (или `/usr/lib64/`). Это позволяет безопасно модифицировать шейдеры, не затрагивая глобальную установку.

Утилита сгенерирует:
* Главный файл загрузки: `~/.config/interactive-wallpaper/init.lua`
* Конфигурации каждого плагина: `~/.config/interactive-wallpaper/plugins/*.lua`

### Симлинки и кастомные профили
Движок полноценно поддерживает символические ссылки. Если вы разрабатываете плагин в отдельной директории, просто создайте симлинк:
```bash
ln -s ~/Projects/my-shader-effect ~/.config/interactive-wallpaper/effects/my-shader-effect
```
Для запуска изолированных профилей (например, "Ночной режим" с другими эффектами) используйте флаг `--config`:
```bash
interactive-wallpaper --config /home/user/my-custom-setup
```

## Запуск

Для полноценной работы интерактивных эффектов необходимо запустить фоновые демоны сбора данных, а затем само ядро. Добавьте следующие строки в автозагрузку вашего оконного менеджера (например, `hyprland.conf`):

```bash
# Запуск демонов сбора данных
audio-daemon &
evdev-pointer-daemon &

# Запуск графического ядра
interactive-wallpaper
```

Динамическое управление параметрами работающего графического ядра осуществляется через утилиту командной строки `shader-desk-ctl`. Для безопасного взаимодействия с эффектами движок использует встроенный вспомогательный Lua-модуль `ctl`. 

Вы можете интегрировать эти команды в конфигурацию вашего оконного менеджера (создать горячие клавиши) или использовать их в пользовательских скриптах:

```bash
# Изменить параметр визуального эффекта для конкретного монитора (например, eDP-1):
shader-desk-ctl "ctl.set('wireframe_mode', false, 'eDP-1')"

# Применить изменение параметра глобально ко всем активным экранам:
shader-desk-ctl "ctl.set('wireframe_mode', false, '*')"

# Переключить логический параметр (True -> False) на всех экранах:
shader-desk-ctl "ctl.toggle('enable_stripes', '*')"
```

Модуль ctl также предоставляет расширенные API для интеграции с внешними инструментами кастомизации (например, чтение палитры из Pywal или реакция обоев на системные уведомления Dunst). Подробное описание команд доступно в технической документации.
