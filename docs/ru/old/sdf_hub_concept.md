# Architectural Concept: SDF Hub & Modular Raymarching

## 1. Введение и проблематика
Текущая архитектура движка основана на концепции изолированных плагинов (`.so`), каждый из которых выполняет независимый Draw Call в собственный (или общий) Framebuffer. Это идеально подходит для классической растеризации (полигоны) и 2D пост-процессинга.

Однако математика Signed Distance Fields (SDF) требует принципиально иного подхода. Для того чтобы два SDF-объекта (например, 3D-текст и сфера) могли взаимодействовать — плавно слипаться (Smooth Minimum), вычитаться или пересекаться — их дистанции должны вычисляться **одновременно, внутри одного цикла Raymarching'а в едином фрагментном шейдере**. 

Данный документ описывает концепцию **"SDF Hub" (Volume Compositor)** — архитектурную надстройку, которая позволит пользователям создавать модульные SDF-плагины, сохраняя при этом возможность их математического взаимодействия и интеграции в общую растеризованную 3D-сцену.

## 2. Основные требования к системе

Будущая реализация должна удовлетворять четырем фундаментальным критериям:

### Требование 1: Корректная интеграция в гибридную 3D-сцену (Z-Buffer)
SDF-объекты не должны выглядеть как "плоская наклейка" поверх или позади растеризованных объектов (полигональных сфер, кубов).
*   **Ожидаемое поведение:** Монолитный объем, сгенерированный SDF-хабом, должен корректно перекрываться полигонами на основе реальной глубины.
*   **Архитектурный вектор:** Движок Raymarching'а должен переводить найденную дистанцию пересечения луча ($t$) в стандартное пространство проекции OpenGL (Clip Space) и осуществлять **запись в `gl_FragDepth`**. 

### Требование 2: Совместимость с конвейером пост-обработки (Blit-shaders)
Хаб должен восприниматься основным ядром (Microkernel) как обычный графический слой.
*   **Ожидаемое поведение:** Пользователь может назначить фильтры (VHS, Dither, Bloom) на результат работы SDF Hub.
*   **Архитектурный вектор:** Hub должен реализовывать или корректно использовать механизмы изоляции FBO (аналог `begin_scaled_pass` / `end_scaled_pass`), чтобы передавать свой отрендеренный слой по цепочке фильтров, прежде чем он попадет в композитор Wayland. Фильтры применяются ко всей слипшейся массе объектов целиком.

### Требование 3: Нативное SDF-взаимодействие (CSG Operations)
Это ключевая фича концепции. Объекты из разных `.so` плагинов должны иметь возможность слипаться.
*   **Ожидаемое поведение:** Пользователь в конфигурации может указать математическую операцию между Нодой А и Нодой Б (add, subtract, intersect, smooth_add с коэффициентом $k$).
*   **Архитектурный вектор:** Реализация паттерна "Uber-Shader". Hub должен динамически запрашивать у подключенных Нод куски их логики (функции дистанции `map` и материалы) и "на лету" собирать из них единый текст фрагментного шейдера перед компиляцией. Способ сборки (строковая конкатенация, AST-парсинг или Node Graph) остается на усмотрение разработчика.

### Требование 4: Экстремальная оптимизация (GPU Efficiency)
Raymarching — алгоритм, тяжело нагружающий ALU видеокарты. Архитектура не должна допускать лишних проходов.
*   **Ожидаемое поведение:** 60/144 FPS на интегрированной и мобильной графике.
*   **Архитектурные векторы (рекомендации):**
    1.  **Single-Pass Raymarching:** Независимо от количества SDF-объектов в Хабе, из камеры выпускается только один луч для каждого пикселя.
    2.  **Early Depth Rejection:** Если луч Raymarching'а находится позади уже существующего в глобальном Z-буфере растеризованного объекта, расчет луча должен немедленно прерываться.
    3.  **Bounding Volumes (Опционально):** Вместо полноэкранного квада Hub может отрисовывать на GPU прокси-геометрию (сферы/кубы), ограничивающую область расчета Raymarching'а только теми пикселями, где объекты реально существуют.
    4.  **Resolution Scaling:** Полная поддержка рендера SDF в пониженном разрешении с последующим апскейлом.

## 3. Концептуальная модель (Разделение ответственности)

Для реализации потребуется разделить сущность "Плагин" на две категории:

### Сущность А: SDF Hub (Оркестратор)
Полноценный плагин (наследник `WallpaperEffect`), который виден главному ядру. 
*   **Зона ответственности:** Инициализация EGL/OpenGL стейта, расчет матриц глобальной камеры, управление циклом `for` для луча (Raymarching engine), расчет освещения сцены, запись `gl_FragDepth` и применение пост-эффектов. 
*   Он не содержит своей геометрии, он только "исполняет" то, что ему дали Ноды.

### Сущность Б: SDF Node (Поставщик данных и логики)
Легковесный интерфейс (рекомендуется создать новый C-ABI, например `ISDFNodeABI`), реализуемый авторами пользовательских плагинов.
*   **Зона ответственности:** 
    *   Экспорт параметров в Lua (цвета, искажения).
    *   Предоставление GLSL-кода формы (функция `d = ...`).
    *   Биндинг своих локальных `uniform` переменных и чтение шины `BlackBoard` (например, для звуковой реакции).
*   *Нода не имеет доступа к FBO, не вызывает `glUseProgram` и не делает Draw Calls.*

## 4. Жизненный цикл (Workflow)

1.  **Конфигурация (Lua):** Пользователь определяет слой `SDF Hub` и передает в него массив дочерних `Nodes` с указанием операций взаимодействия (CSG).
2.  **Инициализация (С++):** Hub загружает Ноды через `PluginManager`, запрашивает их шейдерный код, собирает единый мега-шейдер и компилирует его.
3.  **Render Loop (Hot Path):**
    *   Hub вызывает метод `update()` у всех Нод, позволяя им считать BlackBoard и обновить свои параметры.
    *   Hub биндит единую шейдерную программу.
    *   Каждая Нода биндит свои `uniform` переменные в эту программу.
    *   Hub выполняет один вызов отрисовки (Draw Call), запуская глобальный просчет сцены.

## 5. Открытые вопросы для разработчика
При реализации этой архитектуры будущему инженеру предстоит решить:
*   Как избежать конфликтов имен (Name Mangling) в униформах и функциях, когда Hub склеивает GLSL код из нескольких разных Нод? (Решение: требовать от Нод использовать префиксы, либо делать авто-парсинг и подмену имен на уровне Hub-а).
*   Как интегрировать материалы? Отражения, PBR и текстурирование в Raymarching'е требуют передачи не только дистанции ($t$), но и ID объекта.
