# Рабочее пространство и Разработка шейдеров

Shader Desk предоставляет уникальный Developer Experience (DX) для работы с графикой. Движок берет на себя всю рутину по инициализации EGL-контекстов, перехвату системных событий и управлению памятью, позволяя вам сфокусироваться исключительно на визуальной составляющей — написании GLSL-кода.

В этом разделе описано, как организовать безопасную среду для разработки, как работает механизм мгновенной перезагрузки (Hot-Reload) и как за пару минут перенести эффект с платформы ShaderToy на рабочий стол Wayland.

## 1. Пользовательская песочница (Workspace)

Архитектура Shader Desk построена на принципе **Fallback-приоритетов**. Это означает, что при запросе плагина графическое ядро сначала ищет его в вашей домашней директории, и только если не находит — обращается к системным папкам (например, `/usr/lib/shader-desk`).

Чтобы безопасно модифицировать системные шейдеры или создавать свои, необходимо инициализировать пользовательскую песочницу:

```bash
interactive-wallpaper --init-workspace
```

Эта утилита создаст директорию `~/.config/interactive-wallpaper/effects/`, скопирует туда исходные коды всех доступных в системе плагинов и автоматически внедрит в них скомпилированные бинарные `.so` файлы. 

Теперь вы можете безопасно изменять любые файлы внутри этой папки. Оригинальные системные файлы останутся нетронутыми.

## 2. Анатомия бандла плагина

Каждый визуальный эффект в директории `effects/` — это самодостаточный портативный бандл. Стандартная структура выглядит так:

```text
effects/my-awesome-effect/
├── my-awesome-effect.so    # Нативный C-ABI "драйвер" плагина (скомпилированная логика)
├── presets/                # Готовые Lua-пресеты настроек
│   └── default.lua
└── shaders/                # Исходный код шейдеров
    ├── shader_vert.glsl
    └── shader_frag.glsl
```

Если вы не планируете изменять сложную логику на C++ (например, добавлять генерацию 3D-геометрии), файл `.so` вам трогать не нужно. Вы работаете исключительно с папкой `shaders/`.

## 3. Hot-Reload и Shadow-Commit (Защита от крашей)

Главное преимущество разработки шейдеров в Shader Desk — это бесшовный процесс итераций. Графическое ядро подписано на события файловой системы через системный API `inotify`.

**Как это работает:**
1. Откройте любой `.glsl` файл из вашей песочницы в редакторе кода.
2. Внесите изменения и сохраните файл.
3. Движок мгновенно обнаружит изменение и запустит конвейер рекомпиляции прямо во время работы композитора.

**Механизм Shadow-Commit (Защита от ошибок):**
Вам не нужно бояться опечаток. Если вы допустите синтаксическую ошибку в GLSL-коде, Shader Desk **не зависнет и не выдаст черный экран**. 

Движок использует паттерн "Теневого коммита": при сохранении файла ядро создает временную копию плагина и пытается скомпилировать новые шейдеры в изолированном EGL-контексте. 
* Если компиляция успешна, происходит мгновенный атомарный "свап" (замена старого шейдера на новый). 
* Если в коде ошибка, движок выведет лог компилятора в терминал, молча уничтожит бракованную копию и продолжит плавно отрисовывать старую, рабочую версию ваших обоев.

## 4. Интеграция с ShaderToy (Магия мета-комментариев)

Для многих GLSL-художников стандартом де-факто является платформа ShaderToy. Архитектура Shader Desk позволяет запускать эффекты оттуда практически без модификаций с помощью встроенного плагина `ShaderToy Sandbox`.

Вам достаточно создать новый `.glsl` файл в папке `shaders/` этого плагина и вставить туда код функции `mainImage()`.

Пример файла `shaders/my_city.glsl`:

```glsl
// Это чистый код с ShaderToy. ShaderDesk автоматически создаст вокруг него 
// правильный OpenGL ES 3.0 контекст и пробросит системные переменные.

// =====================================================================
// МАГИЯ АННОТАЦИЙ: ДИНАМИЧЕСКИЕ ПАРАМЕТРЫ ДЛЯ LUA
// Синтаксис: // @param [имя] | [тип] | [значение_по_умолчанию] | [описание]
// =====================================================================

// @param glow_speed | float | 2.5 | Скорость переливания цвета
// @param base_color | vec3 | 0.2, 0.5, 0.8 | Базовый цвет фона
// @param enable_grid | bool | true | Включить геометрическую сетку

void mainImage( out vec4 fragColor, in vec2 fragCoord )
{
    // Стандартные униформы ShaderToy (iResolution, iTime, iMouse) 
    // предоставляются движком автоматически.
    vec2 uv = fragCoord / iResolution.xy;
    uv.x *= iResolution.x / iResolution.y;

    // Используем НАШИ кастомные параметры, объявленные в комментариях выше.
    vec3 col = base_color;

    // Анимация на основе iTime и нашего параметра glow_speed
    col += 0.2 * cos(iTime * glow_speed + uv.xyx + vec3(0, 2, 4));

    if (enable_grid) {
        vec2 grid = fract(uv * 10.0);
        if (grid.x < 0.05 || grid.y < 0.05) col += vec3(0.5); 
    }

    // Реакция на движения мыши (iMouse)
    float distToMouse = distance(uv, iMouse.xy / iResolution.y);
    if (distToMouse < 0.2) col += vec3(1.0, 0.5, 0.0) * (0.2 - distToMouse);

    fragColor = vec4(col, 1.0);
}
```

### 4.1. Стандартные униформы и особенности архитектуры

Движок автоматически предоставляет следующие стандартные униформы ShaderToy: `iResolution`, `iTime`, `iTimeDelta`, `iFrame` и `iMouse`.

**Критическое замечание по переменной `iMouse`:**
Для корректной работы реакции на мышь вы обязаны убедиться, что в вашем конфигурационном файле `init.lua` включен провайдер данных `Evdev Pointer Provider`. Если этот фоновый демон сбора данных отключен или не запущен, переменная `iMouse` всегда будет передавать нулевые значения.

**Особенности Wayland HiDPI и масштабирования (`iResolution`):**
В архитектуре Wayland размер буфера отрисовки зависит от пользовательского масштаба монитора (Scale). Более того, Shader Desk поддерживает оптимизационный параметр `fbo_scale` (рендеринг в пониженном внутреннем разрешении с последующим апскейлом). 

Это означает, что переменная `iResolution` передает фактический размер Framebuffer Object (FBO), который может не совпадать 1:1 с физическими пикселями монитора. Если ваш алгоритм опирается на строгую попиксельную математику (например, `mod(gl_FragCoord.x, 2.0)` для создания паттернов дизеринга), при нестандартном масштабировании результат может выглядеть размыто или искаженно. Старайтесь использовать нормализованные координаты (UV).

### 4.2. Генерация интерфейса управления

Плагин `ShaderToy Sandbox` написан так, что он парсит комментарии `// @param` прямо из исходного кода шейдера перед его компиляцией. 

После сохранения файла выполните команду:
```bash
interactive-wallpaper --init-config
```
Движок прочитает ваши комментарии и автоматически сгенерирует для них Lua-интерфейс в `~/.config/interactive-wallpaper/plugins/`. Теперь вы можете управлять вашим кастомным эффектом из командной строки, привязывать его к музыке или менять цвета через Lua-скрипты.

## 5. Доступ к системным данным (Zero-Latency)

Ваши шейдеры не изолированы от системы. Если в `init.lua` включены Data Providers (демоны аудио или мыши), они непрерывно стримят данные в оперативную память (на шину `BlackBoard`). 

Обычно нативные C++ плагины сами извлекают эти данные и передают их в шейдер как `uniform`. Если вы используете штатные шаблоны или `shadertoy-effect`, вам доступны следующие униформы для создания аудиореактивных эффектов:

```glsl
uniform float audio_bass;      // Уровень низких частот (0.0 - 1.0)
uniform float audio_mid;       // Уровень средних частот
uniform float audio_treble;    // Уровень высоких частот
uniform float audio_bands[64]; // Массив точного спектра для эквалайзеров
```

Просто умножьте масштаб геометрии или интенсивность свечения на `audio_bass`, и ваш шейдер начнет пульсировать в такт музыке из PipeWire без каких-либо задержек.

## 6. Что дальше? Выход за пределы GLSL

Чистый GLSL и ShaderToy невероятно мощны для полноэкранных 2D-эффектов, фракталов и рей-марчинга (Raymarching). 

Однако, если ваша задумка требует:
* Загрузки внешних ресурсов (PNG-текстур, 3D-моделей OBJ/GLTF).
* Управления собственными Ping-Pong буферами (FBO) для сохранения следов, эффектов обратной связи (Feedback Loop) или симуляции гидродинамики (Navier-Stokes).
* Процедурной генерации сложной геометрии (например, Boids-флокинг или симуляция 100 000 частиц), где расчет физики логичнее и эффективнее выполнять на центральном процессоре (CPU).
* Нестандартного взаимодействия с шиной данных `BlackBoard`.

Вам потребуется выйти за рамки GLSL и написать логику "драйвера" плагина на системном языке программирования, который будет напрямую отдавать команды видеокарте через OpenGL API.

### Выбор языка для плагина

Интерфейс плагинов Shader Desk (`IWallpaperEffectABI`) построен на **строгом C-ABI** без использования C++ STL в публичных методах. Это ключевое архитектурное решение означает, что вы можете написать плагин на **любом языке**, который умеет компилироваться в разделяемую библиотеку (`.so`) и экспортировать C-совместимые функции. 

Помимо C++, вы абсолютно свободны использовать:
* **Rust:** используя `#[no_mangle] pub extern "C"` и контейнеры для работы с FFI.
* **Zig:** благодаря его нативной и прозрачной интеграции с C-заголовочными файлами.
* **C:** (чистый стандарт C99/C11).

**Почему по умолчанию мы говорим про C++?**
Официальный SDK, предоставляемый вместе с проектом, предлагает наиболее комфортный Developer Experience именно для современного C++ (C++17/20). Движок поставляется с готовыми C++ "обертками", которые скрывают сырые указатели C-ABI за удобными абстракциями, включает генератор шаблонов кода (boilerplate) и использует библиотеку `GLM` для векторной математики.

Если вы готовы создать свой первый нативный визуальный плагин (используя официальный C++ SDK или адаптируя C-ABI под свой любимый системный язык), переходите к следующему разделу: **[04-sdk-visual-plugins.md](04-sdk-visual-plugins.md)**.
